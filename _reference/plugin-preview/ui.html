Design notes

Why light theme

Plugin panels render inside Figma's own UI, which is light-by-default. Mirroring HiNAS Cloud / Day-mode tokens keeps the panel from looking out of place against Figma's chrome. The HiNAS Blue (#1379F3) primary keeps the panel recognizably ours, not generic-Figma.

Why 360 px wide

Figma plugin panels have a sane max around 400 px before they crowd the canvas. 360 px gives a comfortable two-column feel for the coverage table without forcing horizontal scroll inside the panel.

Why the coverage review is a separate screen

It's the single highest-leverage UX upgrade over the CLI. In the slash command, this table prints to terminal mid-run. Here, the user can edit a coverage decision before any artboard is created. Most runs spend more time on this screen than any other — that's the point.

Why gates overlay rather than route

The DS-conflict modal preserves the build context behind it. The user can see what step we're on, what's already built, and what the question affects. A separate routed screen would lose that context and feel more like an error than a deliberate confirmation.

What's missing on purpose

No render preview of the artboards before build (cost > value in v0.1). No re-run diff (v0.3). No settings panel beyond target page (v0.2 adds API key for Claude judgment steps). No multi-DS picker (v0.4).

Resolved this round

Row interaction → Option C, per-row drill-down (Screen 07). Target page → inline dropdown anchored under the field (Screen 08, 09). Create new page → inline input, no separate modal (Screen 09). Build click → button shows processing state, then panel transitions to Screen 04 (Screen 10). Field affordance → Screen 01 dropdown caret matches Screen 08, no more "Edit" text link.

Still open

Done screen: do we want a collapsible artboard list under the four-line summary, or is the summary enough? Jargon sweep: "DS Candidate", "Pre-flight", "Defaults" still use internal vocabulary — worth a one-pass cleanup before copy is locked.
