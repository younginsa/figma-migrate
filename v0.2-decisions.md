# figma-migrate v0.2 — Architecture Pivot Decisions

This doc captures decisions made during the v0.2 redesign. Read this alongside the mockup (`figma-migrate-plugin-v0.2-mockup.html`).

The mockup shows UI states. This doc covers behavior, rules, and architecture decisions that are not visible in the UI alone.

---

## Core architectural pivot

**v0.1 (deprecated approach):** DS-first matching. Plugin tries to match HTML elements to existing DS components, then renders DS instances directly.

**v0.2 (this redesign):** Pattern-group approach. Plugin detects repeated patterns in HTML and creates its own component+instance structure. DS alignment is done by the designer working on the masters in Figma, not by the plugin matching.

**Why the pivot:**
- DS matching is hard to get right without LLM, and even with LLM has accuracy issues
- Pattern detection is deterministic and reliable
- Failure mode is safer (wrong grouping = drag to fix; wrong DS match = silent bad output)
- Designer effort shifts from "review 50 matches" to "design 10 masters"

---

## What v0.2 does NOT do (explicit non-goals)

- **No DS matching.** Plugin does not call `dsSync()` to read DS components for matching purposes.
- **No DS Candidate detection.** All non-matched elements either become patterns (if repeated) or unique element placeholders (if unique).
- **No LLM integration in v0.2.** Pattern detection is deterministic. LLM is v0.3+ backlog.
- **No coverage map UI.** No `Screen 3 (Review DS mapping)`, `Screen 7 (Edit mapping row)`, `Screen 6 (DS Conflict gate)`, `Screen 11 (Source picker)` from v0.1.
- **No designer code input.** Plugin does not ask designers to write CSS selectors, trigger code, or any kind of code to define states. State capture is either auto-detected or done via visual interaction in the iframe.

---

## Topbar title conventions

Most screens (Screen 1, 1b, 2) use the convention:
- **title** = filename (e.g., `"cpp-setting-tab-mockup.html"`)
- **eyebrow** = screen-specific subtitle

**Pattern Detail (Screen 3) intentionally breaks this:**
- **title** = `"Pattern: {pattern.name}"` (e.g., `"Pattern: btn primary"`)
- **eyebrow** = `"{N} instances across {M} artboards"`

**Rationale:** Pattern Detail is a deep-drill view where the current context (which pattern) matters more than global context (which file). Future deep-drill screens may follow the same asymmetry. Don't try to "normalize" topbar patterns across screens.

---

## Plugin panel sizing

Plugin panel uses **dynamic resize** via `figma.ui.resize(w, h)`:

- **Default size:** 380 × 800 — used for Screen 1, 1a, 1a-fail, 2, 2b, 2c, 4, 5, 6
- **Wide size:** 800 × 800 — used only for Screen 3 (Pattern detail)
- **Extra-wide size:** 1000 × 800 — used only for Screen 1b (Manual snapshot UI). Designer can resize larger if needed.

Plugin calls `figma.ui.resize(w, h)` on navigation in/out of these screens. Leaving the screen restores 380 × 800.

## State detection — boot capture + multi-pattern auto-detect

When the designer clicks "Detect states" on Screen 1, plugin always captures the **boot state** (the page as it loads, before any state-switching). Then it scans the input HTML for known state-marker attributes, in this priority order:

1. **`[data-dev]`** — explicit dev panel markers (v0.1 convention, current fixture's pattern)
2. **`[data-state]`** — common alternative naming
3. **`[data-page]`** — page-marker convention
4. **`[data-view]`** — view-marker convention

If any pattern matches, the detector clicks each found element programmatically (inside a hidden iframe) and captures the resulting DOM. These are *additional* captures on top of the boot capture.

**Key behavior change:** Every successful parse produces at least 1 capture (the boot state). Auto-detect never fully fails — it just produces "boot only" or "boot + N additional states." This matches reality: most mockups don't have state markers, but they still represent at least one screen the designer wants in Figma.

Marker patterns live as a configurable constant at the top of the parser file:

```js
const STATE_MARKER_PATTERNS = [
  '[data-dev]',
  '[data-state]',
  '[data-page]',
  '[data-view]',
];
```

This makes it trivial to add more attribute patterns later. We intentionally do NOT scan for JS `case 'X':` statements or any other code-level patterns — those find state *names* but can't trigger state changes (closure-scoped functions), so they're useless without a clickable marker.

**Return shape:**

```js
{
  bootCapture: Capture,                    // always present
  autoDetect: {
    method: '[data-dev]' | null,           // null if no marker matched
    additionalCaptures: [Capture, ...],    // empty if no marker matched
    scanned: ['[data-dev]', '[data-state]', '[data-page]', '[data-view]']
  }
}
```

**Screen 1a behavior depends on `autoDetect.method`:**
- **`method !== null`** — Auto-detect succeeded. Show "N states detected via {method}" (where N = additionalCaptures.length + 1 for boot). Designer chooses [Continue] or [Customize].
- **`method === null`** — No additional states found. Show "1 state captured (boot only). No state markers detected." Designer chooses [Continue with 1 page] or [Capture more states] (opens Screen 1b in manual-add mode).

The old "Screen 1a-fail" treatment is gone — there's no full-fail case anymore. There's always at least 1 capture.

## Multi-screen mockup limitation (v0.2 scope)

Some mockups present multiple screens in a single static page (e.g., a design doc with "이력 목록" panel next to "이력 상세" panel, no state switcher). v0.2 captures these as **a single boot artboard** — designer must split them manually in Figma.

Region-based area selection (drag a rectangle, capture that subtree as a separate artboard) is **out of scope for v0.2**. It's a v0.3 backlog item because:
- It changes the mental model of "1 capture = 1 page state"
- It adds significant UI complexity (region overlay, selection accuracy, edge cases)
- v0.2's value is provable without it: state-machine mockups work via auto-detect, single-state mockups work via boot capture, multi-screen mockups produce a usable (if not ideal) starting point

## Manual snapshot capture (Screen 1b)

The Snapshot UI handles two cases:

**Case A — Customize from auto-detect (Screen 1a → "Customize" button):**
- Designer arrives at Screen 1b with auto-detected states pre-captured, each marked with an "auto" badge in the gallery.
- Designer can add additional states (e.g., hover variants, modal-open states) that auto-detect missed.
- Designer can delete or rename any captured state.
- "Done" exits Snapshot UI with the final list, proceeds to Screen 2.

**Case B — Pure manual (Screen 1a-fail → "Capture manually" button):**
- Designer arrives at Screen 1b with empty gallery.
- Designer interacts with the iframe (clicks, hovers, navigates) to reach each state they want.
- Designer clicks "Capture this state" — current DOM is added to gallery.
- Repeat until all desired states are captured, then "Done."

**Capture mechanism (both cases):**

```js
function captureFromIframe(iframe, label, source) {
  const doc = iframe.contentDocument;
  return {
    id: crypto.randomUUID(),
    label: label || `State ${captures.length + 1}`,
    html: doc.documentElement.outerHTML,    // for Screen 3 iframe srcdoc later
    elements: walkDom(doc.body),             // for pattern detection
    thumbnail: null,                          // generated async via html2canvas
    source,                                   // 'auto' or 'manual'
    capturedAt: Date.now()
  };
}
```

For **auto-captured** states, plugin uses a hidden iframe and invokes the state-setter programmatically (e.g., `iframe.contentDocument.querySelector('[data-dev=populated]').click()`), waits for DOM to settle (`await sleep(50)`), then calls `captureFromIframe(iframe, stateName, 'auto')`.

For **manual** captures, the iframe is *visible to the designer* and they interact with it directly. Plugin just calls `captureFromIframe(iframe, label, 'manual')` when "Capture this state" is clicked.

**Snapshot gallery rules:**
- Captures listed in chronological order (most recent at bottom).
- Each card shows: thumbnail (small html2canvas screenshot), label (editable on hover via pencil icon), source ("auto" badge if applicable), delete button (visible on hover).
- No deduplication — designer can capture the same DOM twice intentionally (rare, but allowed).
- Captured state can be re-ordered by drag (uses sortablejs).

## Screen 3 — iframe preview behavior

The pattern detail screen shows the **entire HTML page** in an iframe, not extracted CSS or isolated element snippets. The selected instance is highlighted within its full page context.

**Implementation:**

```js
iframe.srcdoc = `
  <style>
    [data-figma-migrate-highlight] {
      outline: 2px dashed #EF4444;
      outline-offset: 3px;
      box-shadow: 0 0 0 1px rgba(239, 68, 68, 0.2);
    }
  </style>
  ${originalPageHtml}
  <script>
    const target = document.querySelector('${instanceSelector}');
    if (target) {
      target.setAttribute('data-figma-migrate-highlight', '');
      target.scrollIntoView({ block: 'center', inline: 'center' });
    }
  </script>
`;
```

**Why this approach over extracted CSS:**
- 100% visual accuracy — original `<style>` blocks, computed styles, all CSS context preserved
- Element appears in its full page context (helpful for designer to understand "where is this used?")
- Implementation is simpler than extraction (no DOM walking, no style merging)

**Trade-offs accepted:**
- Page may be wider than the 800px panel — viewport scrolls horizontally, zoom controls let user adjust
- Each instance row click reloads iframe (new srcdoc) — acceptable since instance count per pattern is small (typically < 20)

**Zoom controls:** Bottom-right corner of iframe — `−` / current zoom % / `+`. Implemented via CSS `transform: scale()` on the page wrapper inside the iframe.

**Fallback:** "Show code" toggle (default OFF). If iframe preview is broken for any reason (e.g., page uses external CDN that fails to load), the designer can toggle code view to see the raw HTML snippet.

## Output model

For each HTML page in the input, plugin produces:
- One Figma artboard
- Optionally, a screenshot of the HTML page as background reference layer (settings toggle)

For each repeated pattern (≥ N occurrences, default N=2):
- A Figma component (master) on the "Components (generated)" page
- N instances placed in their respective artboards at the original HTML positions

For each unique element (only 1 occurrence):
- A placeholder box at the original position
- Inside the box: label according to "Detail level" setting (simple / detail)

For each group of patterns the designer merged:
- One Figma component set (`figma.combineAsVariants()`)
- Each member pattern becomes a variant within the set

## Pattern detection rules (deterministic)

Two signals:
1. **Structural hash:** Recursive hash of `tagName + children-structure`. Two elements share a structural hash if their DOM trees are identical in shape (tags + nesting), regardless of class names or content.
2. **Class string:** The exact `class` attribute value.

Default detection rule (configurable via Settings):
- Same structural hash AND same class string → same pattern
- Same structural hash AND different class string → different patterns (separate)
- Different structural hash → different patterns

Settings toggles:
- **Match by class name** (default ON): include class string in pattern key
- **Match by structure** (default ON): include structural hash in pattern key
- **Split on class modifiers** (default ON): treat `btn-primary` and `btn-danger` as separate patterns. When OFF, only the base class (`btn`) matters.
- **Minimum instances** (default 2): pattern key must occur ≥ N times. Below threshold → unique element placeholder.

## Screen 2 — pattern list ordering

Groups pin to the TOP of the pattern list, above all ungrouped patterns. Each section is sorted by count descending within itself:

1. **Groups** (sorted by `totalCount` desc)
2. **Ungrouped patterns** (sorted by `count` desc)

**Rationale:** Groups are the designer's deliberate work — they should be prominent. This overrides an earlier Phase 7 decision (Q1) that interleaved groups with ungrouped patterns in a single count-sorted list.

---

## Grouping (variants) — designer-driven

Plugin does NOT auto-group patterns into variant sets. Designer explicitly groups them via the selection mode.

**Selection mode flow:**
1. Designer clicks "Select to group →" on Screen 2
2. Checkboxes slide in, drag handles appear on hover (left of checkbox)
3. Designer taps 2+ patterns to select
4. Footer shows "Merge selected (N) →" button
5. Click → patterns become a group with auto-generated name (longest common prefix of class names, e.g., "btn" from btn-primary + btn-secondary; falls back to "Group 1" if no common prefix)
6. Designer can edit group name (pencil icon)
7. Drag patterns in/out of groups to refine
8. Group selection: click group header → shows unlink icon on right → click unlink to disband group
9. "Done" exits selection mode

**Critical UX rules:**
- Items inside a group can only be selected with other items in the same group (cross-group/ungrouped selection disabled while a group item is selected)
- Cross-group movement uses drag-and-drop ONLY (not checkboxes)
- Drag out = item returns to ungrouped position in original list
- No nested groups (groups cannot contain groups)
- Group merging: dragging one group onto another merges all members into a single set

## When component sets get created (Figma side)

**NOT at grouping time.** Plugin maintains group state in memory only until build.

**At build time:**
- For each standalone pattern (not in a group): `figma.createComponent()` → name = pattern name
- For each group of patterns: create each member as a component, then `figma.combineAsVariants(members, parent)` → name = group name
- Variant property name defaults to "Variant"; values derived from pattern's distinguishing class modifier (e.g., `btn-primary` → "primary")
- All masters live on the "Components (generated)" page

This separation matters: the designer can freely experiment with grouping in the plugin without polluting the Figma canvas. Only "Build" makes it permanent.

---

## Persistence

Per-file settings (targetPage / componentsPage / createIntents / build-time toggles) live in **`figma.root` pluginData**, NOT in `clientStorage`. Reason: `clientStorage` is per-user-global and leaks across files; `figma.fileKey` requires private-API publishing, so it can't be used to scope `clientStorage` keys per file either. `clientStorage` is reserved for true user-global preferences only. (Corrected in Phase 9a after observing a components-page leak across files.)

Storage key: `'figma-migrate:v0.2:settings'` written/read via `figma.root.setPluginData / getPluginData` (synchronous; no `await`). Corrupt JSON is treated as fresh.

Settings persisted:
- `targetPage` and `componentsPage` (each is `{id, name}` for existing pages, or `{create: true, name}` for pending-create intents)
- `componentizeThreshold` (default 5; wired UI in Phase 12)
- `addScreenshotBackground` (default false; wired UI in Phase 12)

Group state is NOT persisted — each parse run starts fresh.

---

## Library dependencies

- **sortablejs** for drag-and-drop. Don't roll your own. 45KB, no other deps, well-tested.
- **html2canvas** (already in manifest from v0.1) for screenshot background option.
- No LLM/AI libraries. No HTTP calls to external services.

---

## Build pipeline phases (Screen 4 stepper)

1. Parse HTML — split into pages, build DOM tree per page
2. Detect patterns — apply detection rules, classify elements
3. Create variants — for groups: create component set + variants; for standalone patterns: create single component
4. Build artboards — one frame per HTML page, sized to page content
5. Place instances — instantiate components/variants at HTML positions
6. Place unique element placeholders — draw labeled boxes for unique elements

Each phase reports progress to UI for the Screen 4 stepper.

---

## Detail level (unique element label style)

For unique elements rendered as placeholder boxes:
- **simple**: just element type, e.g., `"input"`, `"div"`, `"a"`
- **detail**: type + first attribute hint, e.g., `"input · email"`, `"a · href"`, `"div · profile-card"`

Default: detail.

---

## Migration plan from v0.1

Recommend keeping v0.1 in a separate branch / folder. Don't refactor v0.1 in place — too many things change.

Phased build order:
1. Set up new project directory with manifest pointing to new code.js / ui.html
2. Port HTML parser from v0.1 (the existing `parseHtml` is reusable for the structural side)
3. Remove DS sync entirely (or stub it for v0.3)
4. Add pattern detection logic (new code, ~200 LOC)
5. Wire Screen 1 (Paste HTML) → Screen 2 (Patterns detected)
6. Add selection mode (Screen 2b/2c) with sortablejs
7. Add Screen 3 (pattern detail with iframe preview)
8. Build pipeline (variants creation, artboard placement, unique element placeholders)
9. Settings screen
10. Smoke test against real Avikus HTML mockups

---

## Out of scope for v0.2 (parking lot)

- LLM-based pattern suggestion (e.g., "these 3 look like variants, want to group them?")
- Visual fingerprint matching (similar-looking elements, not just structural)
- DS sync for `combineAsVariants()` to suggest existing DS variants for matched groups
- Multi-file workflow (apply same patterns across multiple HTML files)
- Diff mode (re-run on updated HTML, only rebuild changed parts)

## v0.3 backlog

### Filter dev-panel scaffolding from pattern detection

Mockups often include developer/testing scaffolding (e.g., `cpp-setting-tab-mockup.html` has a "Demo: Populated ›" state-switcher toolbar with `.dev-panel__btn` buttons). These are not real product UI and should not appear as detected patterns.

**Current behavior:** dev panel elements appear in pattern list (e.g., `dev panel__btn` at count 99 in cpp fixture).

**Proposed fix:** parser-level filter for class prefixes specified in Settings. Default ignore list: `dev-panel`, `dev-tools`. Settings UI allows designer to add/remove prefixes per-project.
