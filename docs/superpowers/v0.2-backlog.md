# Figma-migrate plugin — v0.2 backlog

Items surfaced during v0.1 development that were intentionally deferred. Each represents a meaningful chunk of work, not a polish fix.

---

## 1. Auto-capture visual previews for unmapped DS Candidates

**Status:** Surfaced during Phase D verification (2026-05-18)

**Current behavior:** Unmapped DS Candidates render in the build output as labeled placeholder frames — text-only, no visual.

**Desired behavior:** Every `data-ds-candidate` element should get a visual preview in its artboard, even without a user-picked DS mapping.

**Approach:**
- During parse-html, render the user's HTML in a hidden iframe
- For each `data-ds-candidate` element, html2canvas-capture it
- Attach the captured PNG bytes to the candidate object in the parsed payload
- Build pipeline embeds the captured image as the artboard background when no DS mapping exists (use existing `figma.createImage` infrastructure)

**Risks / unknowns:**
- HTML's JS must execute for state-dependent content to render — iframe sandboxing limits
- Large mockups may be slow (~30 candidates × 200ms each)
- Some elements (charts, custom canvas) may not render via html2canvas

**Estimate:** 2-3 hours focused work + debug

## 2. Per-element DS component instantiation in state artboards

**Status:** Surfaced during Phase D verification (2026-05-18, "input field detection" issue)

**Current behavior:** `parseHtml` detects `<input>`, `.btn`, etc. and lists them in "DS components used" on Screen 02 — but the build pipeline does NOT instantiate them into state artboards. The Setting dialog body is left as a scaffold for the user to fill in.

**Desired behavior:** When parser detects a `<input>` in the "edit" state HTML, the build should instantiate the DS Input component inside the Setting dialog body of the `Setting - CCP - Edit` artboard.

**Approach:**
- Map detected `dsComponents` items to DS_KEYS where possible (input → DS_KEYS.Input, .btn → DS_KEYS.Button, etc.)
- For each state's artboard, walk the HTML for that state's content and instantiate the corresponding DS components in the right positions
- Requires HTML→Figma layout translation (currently absent)

**Risks / unknowns:**
- Positioning is non-trivial (HTML uses CSS flexbox/grid; Figma uses auto-layout — translation is lossy)
- Multiple instances of same component need separate keys
- May produce visually wrong layouts without significant work

**Estimate:** 1-2 weeks, larger than expected

## 3. Multi-library / team-library support

**Status:** Investigated during Phase C, deferred

**Current behavior:** Plugin requires a "Design system" page in the same Figma file. Team libraries are not supported.

**Desired behavior:** User can pick any team library as the DS source.

**Blocker:** `figma.teamLibrary.getAvailableLibrariesAsync()` either doesn't exist publicly or doesn't enumerate currently-enabled libraries in community plugins. The Figma plugin API doesn't expose what the Assets panel shows.

**Possible paths:**
- Use Figma's REST API (separate from Plugin API) — requires OAuth, network access, server, big scope
- Wait for Figma to expose library enumeration in Plugin API
- Manual user input: paste a Figma library file URL/key, use `importComponentByKeyAsync` for known component keys

**Estimate:** Significant — depends on path chosen

## 4. Resume after panel close mid-build

**Status:** Spec §5.2 originally requested, deferred for v0.1

**Current behavior:** If user closes the plugin during a build, the in-progress build is abandoned. Reopening starts from scratch.

**Desired behavior:** Reopening detects in-progress state via `figma.clientStorage`, offers a "Resume build" option.

**Approach:**
- Checkpoint build progress to clientStorage between artboards
- On plugin open, check for stale build state, prompt user
- Resume from last checkpoint by skipping already-created artboards (by name lookup)

**Estimate:** ~4 hours

## 5. DS conflict gate (Screen 06)

**Status:** Originally Phase B Gap 2, deferred for v0.1

**Current behavior:** Screen 06 has UI markup (Pause build / Apply & continue) but no wiring. code.js never emits a `conflict` message.

**Desired behavior:** When code.js detects a DS component renders wrong in context (e.g., dark instance in light file, missing variant), it pauses the build and prompts the user.

**Blocker:** Defining what "conflict" programmatically means. Original brainstorming flagged this as needing design specification before implementation.

**Estimate:** 1 day spec + 1 day implementation
